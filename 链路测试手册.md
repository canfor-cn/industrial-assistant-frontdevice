## 测试环境准备

### 1. 依赖安装

确保已安装所有依赖：

```bash
# 激活对应的Conda环境
conda activate wakefusion  # 或 wakefusion_vision（视觉模块）

# 安装依赖
pip install -r requirements.txt
```

### 2. 配置文件检查

确认 `config/config.yaml` 包含以下配置段：

```yaml
zmq:
  vision_pub_port: 5555
  audio_pub_port: 5556
  audio_ctrl_port: 5557
  asr_pull_port: 5558
  asr_result_push_port: 5562
  vision_ctrl_pub_port: 5564
  core_control_rep_port: 5561

vad:
  enabled: true
  engine: "silero"
  threshold: 0.5
  sample_rate: 16000

# ASR配置（已迁移到服务器端，配置已注释）
# asr:
#   enabled: true
#   engine: "funasr"
#   model_name: "paraformer-zh-online"
#   use_cache: true

# TTS配置（已迁移到服务器端，配置已注释）
# tts:
#   enabled: true
#   engine: "qwen3-tts"
#   model_name: "Qwen3-TTS-12Hz-0.6B-Base"
#   ref_audio_path: "path/to/reference_voice.wav"
#   warmup_enabled: true

audio_playback:
  sample_rate: 16000  # 采样率（Hz），必须与服务器端TTS输出一致
  # 重要提示：
  # - 如果服务器端使用Qwen3-TTS，其默认输出为24000Hz，此处应设置为24000
  # - 如果服务器端使用其他TTS引擎输出16000Hz，则设置为16000
  # - 使用Mock测试脚本时，脚本中的SAMPLE_RATE也必须与此配置保持一致
  format: "pcm_int16"
  channels: 1
  prebuffer_ms: 100  # 预缓冲时长（毫秒）

llm_agent:
  host: "127.0.0.1:8080"  # LLM Agent服务地址
  device_id: "wakefusion-device-01"
  token: "your-token-here"
  use_ssl: false
  reconnect_interval_sec: 5.0
  ping_interval_sec: 30.0

vision_wake:
  detection_distance_m: 4.0  # 4米内开始检测
  frontal_percent_threshold: 70  # 正面率阈值（%）
  distance_range: [0.4, 4.5]  # 有效距离区间（米），用于视觉斩断判断
  enter_debounce_ms: 200  # 进入防抖窗口（毫秒）
  leave_debounce_ms: 1000  # 离开防抖窗口（毫秒）
  visual_cutoff_enabled: true

audio_threshold:
  default: 0.9  # 默认高阈值（无人时，纯语音唤醒模式）
  visual_wake: 0.6  # 视觉唤醒后的低阈值（视觉降维打击模式）

conversation:
  vad_silence_timeout_default_sec: 5.0  # 宏观超时（秒）：纯语音唤醒模式下，5秒无语音则结束对话
  vad_silence_timeout_visual_sec: 5.0  # 宏观超时（秒）：视觉降维打击模式下，5秒无语音且无唇动则结束对话
  vad_fast_cutoff_sec: 1.5  # 双模态快刀超时（秒）：连续1.5秒无声音且无唇动则结束对话
  interactive_timeout_sec: 90.0  # 交互超时（秒）：90秒无交互则退出交互模式
```

### 3. 硬件检查

- **摄像头**：确保Orbbec Femto Bolt已连接并驱动正常
- **麦克风**：确保音频输入设备可用（XVF3800或默认设备）
- **模型文件**：
  - 唤醒词模型：`xiaokang_xvf3800_pro.nemo` 或 `xiaokang_oww.onnx`
  - TTS参考音频：`path/to/reference_voice.wav`（Qwen3-TTS Voice Clone必需，几秒钟WAV格式）

### 4. 环境检查

确保三个Conda环境已正确配置：

- **wakefusion_vision**：opencv-python, mediapipe, pyyaml, pyzmq, pydantic, numpy
- **wakefusion**：torch (CPU版), torchaudio (CPU版), sounddevice, soundfile, pyyaml, pyzmq, pydantic, numpy
- **wakefusion_cosyvoice**：torch (GPU版), torchvision, torchaudio, transformers, funasr, modelscope, websockets, qwen-tts, pyyaml, pyzmq, pydantic, numpy

### 测试1：视觉模块独立测试

**目的**：验证视觉模块的ZMQ PUB输出和视觉唤醒状态机

**步骤**：

1. **启动视觉模块**（在 `wakefusion_vision` 环境中）：
   ```bash
   conda activate wakefusion_vision
   python -m wakefusion.services.vision_service --fps 15
   ```

2. **在另一个终端启动测试脚本**：
   ```bash
   conda activate wakefusion
   python wakefusion/tests/test_vision_sub.py
   ```

3. **测试场景**：
   - **场景A：视觉唤醒**
     - 走到摄像头前3米内
     - 正脸看摄像头（正面率 >= 75%）
     - **预期**：测试脚本应显示 `wake: true`
   
   - **场景B：视觉唤醒结束**
     - 从摄像头前走开，距离超过3.5米
     - 等待约1.5秒
     - **预期**：测试脚本应显示 `wake: false`

4. **验证点**：
   - ✅ ZMQ连接成功
   - ✅ 收到包含`wake`字段的JSON数据
   - ✅ 视觉唤醒状态正确切换
   - ✅ 距离和正面率数据正确

**预期输出示例**：
```
正在监听视觉模块数据...
走到摄像头3米内，正脸看它，应该看到 wake: true
走开3.5米外，1.5秒后应该看到 wake: false
------------------------------------------------------------
[14:30:15] 视觉唤醒状态变化: True
  -> 检测到人脸: 1 个
  -> 距离: 2.5m
[14:30:20] 视觉唤醒状态变化: False

📝 [中间结果] 你

------------------------------------------------------------
📝 [中间结果] 你好

------------------------------------------------------------
✅ [最终结果] 你好
   置信度: 0.95 | 时间戳: 13:18:35 | 序号: #3
------------------------------------------------------------
```

**WebSocket消息格式**（ASR → LLM）：
```json
{
  "type": "asr_result",
  "text": "你好",
  "is_final": false,  // false=中间结果（流式识别），true=最终结果
  "confidence": 0.85,
  "timestamp": 1234567890.123
}
```

**WebSocket接入信息（供LLM接入）**：
- **连接地址**：`ws://127.0.0.1:8766`（本地）或 `ws://<服务器IP>:8766`（远程）
- **端口**：`8766`（可在 `config/config.yaml` 中配置 `websocket.asr_port`）
- **协议**：WebSocket (ws://)
- **消息类型**：`asr_result`
- **字段说明**：
  - `type`: 固定为 `"asr_result"`
  - `text`: 识别出的文本内容
  - `is_final`: `false` 表示中间结果（流式识别），`true` 表示最终结果
  - `confidence`: 置信度（0.0-1.0）
  - `timestamp`: Unix 时间戳（秒）

**测试工具**：
- HTML测试页面：`tests/test_asr_websocket_client.html`（可视化显示识别结果）
- Python测试脚本：`wakefusion/tests/test_asr_module.py`（发送测试音频）

## 故障排查指南

### 问题1：ZMQ连接失败

**症状**：
- 测试脚本显示 `Connection refused` 或 `Address already in use`

**排查步骤**：
1. 检查端口是否被占用：
   ```bash
   # Windows
   netstat -ano | findstr :5555
   
   # Linux/Mac
   lsof -i :5555
   ```

2. 确认模块启动顺序：
   - PUB端（vision/audio）必须先启动
   - SUB端（core_server）后启动

3. 检查防火墙设置

**解决方案**：
- 关闭占用端口的进程
- 修改配置文件中的端口号
- 检查防火墙规则

---

### 问题2：视觉唤醒状态不正确

**症状**：
- `wake`状态不切换，或切换不及时

**排查步骤**：
1. 检查距离计算：
   - 确认深度相机正常工作
   - 检查`distance_m`字段是否合理

2. 检查正面率计算：
   - 确认人脸检测正常
   - 检查`frontal_percent`字段是否 >= 75

3. 检查离开检测：
   - 确认`leave_check_frames`配置合理
   - 检查`_leave_frame_count`是否正确递增

**解决方案**：
- 调整`detection_distance_m`和`frontal_percent_threshold`
- 调整`leave_timeout_sec`和`leave_check_frames`

4. 检查VAD：
   - 确认VAD检测到语音
   - **如果使用Silero VAD**：检查VAD引擎是否正确初始化
   - **如果使用RMS阈值**：检查RMS阈值是否合理（已废弃）

**解决方案**：

**症状**：

- 持续对话期视觉斩断不触发

### 测试11：延迟测试

**目的**：测量各环节的延迟

**步骤**：

1. **视觉唤醒延迟**：
   - 记录从走到摄像头前到`wake: true`的时间
   - **目标**：< 200ms（@15fps）

2. **音频唤醒延迟**：
   - 记录从说唤醒词到进入LISTENING的时间
   - **目标**：< 500ms

3. **视觉斩断延迟**：
   - 记录从离开摄像头到状态切换的时间
   - **目标**：< 100ms

4. **VAD超时精度**：
   - 记录实际超时时间与配置时间的差异
   - **目标**：误差 < 200ms

5. **ASR首字延迟（TTFB）**：
   - 记录从开始说话到收到第一个识别结果的时间
   - **目标**：< 500ms（流式识别）

6. **TTS首字延迟（TTFB）**：
   - 记录从LLM发送文本到开始播放音频的时间
   - **目标**：< 300ms（标点切分 + 预热后）

7. **硬打断延迟**：
   - 记录从触发唤醒到TTS停止的时间
   - **目标**：< 100ms

## 注意事项

1. **环境隔离**：
   - 视觉模块：`wakefusion_vision` 环境
   - 音频模块和Core Server：`wakefusion` 环境（CPU版PyTorch，用于Silero VAD）
   - ASR模块和TTS模块：`wakefusion_cosyvoice` 环境（GPU版PyTorch，用于FunASR和Qwen3-TTS）

2. **启动方式**：
   - **推荐**：使用统一启动器 `launcher.ps1`（在Anaconda PowerShell Prompt中运行）
   - **调试**：手动启动各个服务（用于单独调试某个服务）

3. **启动顺序**：
   - 启动器会自动按正确顺序启动服务
   - 手动启动时，建议先启动数据生产者（vision/audio），再启动消费者（core_server）
   - ASR和TTS模块可以在Core Server之前或之后启动（独立服务）

4. **端口冲突**：如果端口被占用，修改`config.yaml`中的端口配置

5. **日志级别**：可以通过修改日志级别获取更详细的调试信息

6. **资源清理**：测试结束后记得关闭所有模块，释放资源（使用启动器选项2或关闭服务窗口）

6. **Silero VAD**：
   - 首次加载需要从torch.hub下载模型，可能需要一些时间
   - 确保CPU版PyTorch已正确安装（`wakefusion`环境）
   - 如果VAD检测不准确，可以调整`vad.threshold`配置（默认0.5）

7. **音频流接收**：
   - TTS已迁移到服务器端，通过WebSocket二进制帧接收
   - 确保`audio_playback.sample_rate`配置与服务器端TTS输出一致
   - 默认16000Hz，如果服务器使用Qwen3-TTS（24000Hz），应设置为24000
   - 预缓冲机制（100ms）用于抑制网络抖动

8. **ASR和TTS**：
   - ASR和TTS已迁移到服务器端（LLM Agent），客户端不再需要启动这两个模块
   - 客户端通过WebSocket发送音频流（带traceID）给服务器端进行ASR识别
   - 服务器端通过WebSocket发送TTS音频流（带session_id）给客户端播放

9. **视觉斩断机制**：
   - 视觉斩断触发条件：is_vision_target_present=False（所有人脸都不在[0.4m, 4.5m]区间内，经过1秒防抖窗口）
   - 断尾处理：如果存在未完成的对话（_current_trace_id不为None），先发送audio_end消息闭合ASR管道，再发送timeout_exit
   - 播放状态下的视觉斩断：发送client_interrupt消息，清空队列，生成新session_id

10. **TraceID和SessionID管理**：
    - traceID：在VAD从False变True时生成，用于标记每轮对话
    - session_id：在收到第一个TTS音频帧时生成，用于过滤旧音频帧
    - 硬打断或视觉斩断时会生成新的session_id，屏蔽旧session的音频帧

11. **纯事件驱动架构**：
    - VAD超时检查已改为纯事件驱动，在每个音频帧到达时检查，无需独立线程
    - 优势：避免多线程锁竞争，特别是对_current_trace_id的读写
    - 实时性更好：每个音频帧（20ms或100ms）都会检查，响应更快

12. **90秒交互超时**：
    - 每次VAD由True变False时，重置并重新开始90秒倒计时
    - 90秒后自动退出交互模式，恢复默认阈值，允许立即再次唤醒

13. **硬打断机制**：
    - 硬打断后清空播放队列和预缓冲区
    - 生成新的session_id，屏蔽旧session的音频帧
    - 发送client_interrupt消息给LLM Agent
    - 启动冷却期，期间忽略VAD和KWS事件

**重要更新**：系统已升级为统一WebSocket协议，WakeFusion作为客户端连接LLM Agent服务端。

- **8080端口**：LLM Agent作为WebSocket服务端，WakeFusion（Core Server）作为客户端连接

- **5561端口**：Core Server作为ZMQ REP端，LLM作为REQ端连接，发送控制指令（可选）

**连接地址示例**：
- LLM Agent WebSocket: `ws://127.0.0.1:8080/api/voice/ws?deviceId=wakefusion-device-01&token=your-token`
- Core Server控制: `tcp://127.0.0.1:5561`（本地）或 `tcp://<服务器IP>:5561`（远程）

- ~~8766~~：原ASR WebSocket端口（已废弃）

## WebSocket 接口文档（供LLM接入）

### 统一WebSocket协议（WakeFusion ↔ LLM Agent）

**重要更新**：系统已升级为统一WebSocket协议，WakeFusion作为客户端连接LLM Agent服务端。ASR和TTS已迁移到服务器端，不再使用独立的WebSocket端口。

**连接信息**：
- **地址**：`ws://127.0.0.1:8080/api/voice/ws?deviceId=wakefusion-device-01&token=your-token-here`
- **端口**：`8080`（可在 `config/config.yaml` 中配置 `llm_agent.host`）
- **协议**：WebSocket (ws:// 或 wss://)
- **方向**：WakeFusion（客户端）↔ LLM Agent（服务端）

### 客户端发送的消息（WakeFusion → LLM Agent）

#### 1. audio_start - 开始音频流

**消息格式**：
```json
{
  "type": "audio_start",
  "trace_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "timestamp": 1234567890.123,
  "sample_rate": 16000
}
```

**说明**：
- 在生成新traceID时发送（VAD从False变True时）
- traceID用于标记每轮对话，用于LLM Agent的精确对话定位

#### 2. 二进制音频流（PCM数据）

**消息格式**：WebSocket Binary Frame（PCM int16格式）

**说明**：
- 每个音频帧（20ms或100ms）直接发送，无需攒批
- 必须与audio_start消息绑定（通过traceID）
- 在VAD为False时也继续发送（无traceID），直到触发超时

#### 3. audio_end - 结束音频流

**消息格式**：
```json
{
  "type": "audio_end",
  "trace_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "reason": "vad_timeout"  // 或 "visual_cutoff", "timeout"
}
```

**说明**：
- 在VAD超时或视觉斩断时发送
- reason字段说明结束原因

#### 4. client_interrupt - 客户端打断

**消息格式**：
```json
{
  "type": "client_interrupt",
  "reason": "barge_in"  // 或 "visual-cutoff"
}
```

**说明**：
- 硬打断（唤醒词打断）或视觉斩断时发送
- 服务器端应停止ASR识别和TTS合成

#### 5. device_state - 设备状态上报

**消息格式**：
```json
{
  "type": "device_state",
  "state": "idle"  // 或 "listening", "thinking", "speaking"
}
```

**说明**：
- 在状态切换时自动上报
- 用于服务器端了解客户端状态

#### 6. timeout_exit - 超时退出

**消息格式**：
```json
{
  "type": "timeout_exit",
  "device_id": "wakefusion-device-01",
  "timestamp": 1234567890.123
}
```

**说明**：
- 在视觉斩断时发送（在发送audio_end之后）

### 服务端发送的消息（LLM Agent → WakeFusion）

#### 1. route - TTS文本（仅用于状态上报）

**消息格式**：
```json
{
  "type": "route",
  "text": "今天天气很好，适合出门散步",
  "isFinal": false
}
```

**说明**：
- 仅用于状态上报（如果当前是PROCESSING状态，上报thinking状态）
- 实际的TTS合成在服务器端完成

#### 2. 二进制音频流（PCM数据）

**消息格式**：WebSocket Binary Frame（PCM int16格式，带session_id）

**说明**：
- TTS合成的音频流
- 客户端通过session_id过滤旧音频帧（硬打断或视觉斩断后）

#### 3. TTS_DONE - TTS合成完成

**消息格式**：
```json
{
  "type": "TTS_DONE"
}
```

**说明**：
- TTS合成完成，客户端等待音频队列排空后继续交互模式

#### 4. stop_tts - 停止TTS合成

**消息格式**：
```json
{
  "type": "stop_tts"
}
```

**说明**：
- 服务器端主动停止TTS合成
- 客户端清空播放队列和预缓冲区

#### 5. interrupt - 服务端打断指令

**消息格式**：
```json
{
  "type": "interrupt",
  "session_id": "yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy"
}
```

**说明**：
- 服务器端主动打断TTS播放
- 如果当前未在播放TTS（is_playing_tts=False），直接忽略

### Core Server 控制接口（LLM → Core Server）

**连接信息**：
- **ZMQ REP地址**：`tcp://127.0.0.1:5561`
- **端口**：`5561`（可在 `config/config.yaml` 中配置 `zmq.core_control_rep_port`）
- **协议**：ZMQ REQ-REP

**消息格式**（LLM发送给Core Server）：
```json
{
  "command": "extend_window",
  "value": 15.0
}
```

**支持的命令**：
- `extend_window`: 延长免唤醒窗口（参数：`value` = 秒数）
- `play_video`: 播放视频（未来扩展）
- `set_parameter`: 设置其他系统参数（未来扩展）

**Python接入示例**：
```python
import zmq
import json

context = zmq.Context()
req_socket = context.socket(zmq.REQ)
req_socket.connect("tcp://127.0.0.1:5561")

# 发送延长窗口指令
request = {
    "command": "extend_window",
    "value": 15.0
}
req_socket.send_json(request)
response = req_socket.recv_json()
print(response)  # {"status": "ok", "new_timeout": 15.0}
```
