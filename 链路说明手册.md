## 系统架构概述

WakeFusion是一个多模态唤醒和交互系统，整合音频、视觉、ASR、TTS等模块，实现智能数字人对话。

### 模块组成

- **音频模块** (audio_service.py): 麦克风采集、KWS唤醒词检测、VAD语音活动检测
- **视觉模块** (vision_service.py): 摄像头采集、人脸检测、距离估算、唇动检测
- **核心服务器** (core_server.py): 多模态融合决策、状态管理、音频流控制（瘦客户端架构）
- **LLM Agent服务器** (外部): 包含ASR和TTS功能，通过WebSocket与Core Server通信

### 通信协议

- **ZMQ PUB-SUB**: 数据流传输（视觉数据、音频数据、视觉控制）
- **ZMQ REQ-REP**: 控制指令（阈值调整、流控制）
- **WebSocket**: 统一通信协议（Core Server ↔ LLM Agent，双向：音频流、ASR结果、TTS音频流、设备状态、打断控制）

## 系统状态管理

### 状态标志位（布尔标志位架构）

系统采用布尔标志位替代传统的状态枚举，简化状态管理：

1. **is_interactive_mode**: 是否处于持续对话交互期（替代原来的 LISTENING/PROCESSING 状态）
2. **is_playing_tts**: 当前喇叭是否在出声（替代原来的 SPEAKING 状态）
3. **is_vision_target_present**: 视觉区间[0.4m, 4.5m]内是否有人（用于视觉斩断判断）

### 状态逻辑描述

```
空闲状态 (is_interactive_mode=False, is_playing_tts=False)
  ├─[视觉唤醒]→ 视觉降维打击激活（阈值降至0.6）
  │              ├─[用户说唤醒词]→ 进入交互模式 (is_interactive_mode=True)
  │              └─[用户离开]→ 恢复高阈值 (0.9)
  │
  ├─[纯语音唤醒]→ 进入交互模式 (is_interactive_mode=True)
  │
交互模式 (is_interactive_mode=True, is_playing_tts=False)
  ├─[VAD超时]→ 等待LLM回复（is_interactive_mode保持True）
  ├─[视觉斩断]→ 退出交互模式 (is_interactive_mode=False)
  ├─[90秒超时]→ 退出交互模式 (is_interactive_mode=False)
  └─[30秒保底]→ 强制截断
  │
播放状态 (is_interactive_mode=True, is_playing_tts=True)
  ├─[播放完成]→ 继续交互模式 (is_playing_tts=False, is_interactive_mode=True)
  ├─[硬打断]→ 继续交互模式 (is_playing_tts=False)
  └─[视觉斩断]→ 退出交互模式 (is_interactive_mode=False, is_playing_tts=False)
```

## 核心机制详解

### 1. 视觉唤醒机制

**触发条件**:
- 检测到人脸距离 ≤ 4.0米
- 人脸正面率 ≥ 70%
- 同时满足以上两个条件

**状态维护**:
- 未唤醒: 持续检查是否满足唤醒条件
- 已唤醒: 检查所有人脸是否在[0.4m, 4.5m]区间内（用于视觉斩断判断）
- 离开判定: 防抖窗口（默认1000ms）后，所有人脸都不在有效区间，则取消唤醒

**实例**:
- 用户走到摄像头前3.5米，正面看摄像头（正面率75%）
- 预期: 视觉唤醒状态激活，音频阈值降至0.6
- 用户后退到5米外，持续1秒（防抖窗口）
- 预期: 视觉唤醒状态取消，音频阈值恢复至0.9

### 2. 音频唤醒机制

系统支持两种唤醒路径：

#### 路径A: 视觉降维打击

**触发条件**:
- 视觉唤醒已激活（距离≤4m，正面率≥70%）
- 唤醒词置信度 ≥ 0.6（visual_wake阈值）

**流程**:
1. 视觉模块检测到用户（距离≤4m，正面率≥70%）
2. Core Server发送降阈指令（0.9→0.6），视觉降维打击激活
3. 用户说唤醒词"你好小康"
4. 音频模块检测到唤醒词，置信度≥0.6
5. Core Server进入交互模式（is_interactive_mode=True）

**实例**:
- 用户在摄像头前说"你好小康"（置信度60%）
- 预期: 系统进入LISTENING状态，开始接收音频

#### 路径B: 纯语音唤醒

**触发条件**:
- 不在交互模式（is_interactive_mode=False）
- 视觉唤醒未激活（距离>4m 或 正面率<70%）
- 唤醒词置信度 ≥ 0.9（default阈值）

**流程**:
1. 用户不在摄像头前（或距离>4m，或正面率<70%）
2. 用户说唤醒词"你好小康"，置信度≥0.9
3. Core Server直接进入交互模式（is_interactive_mode=True）

**实例**:
- 用户在摄像头外说"你好小康"（置信度96%）
- 预期: 系统进入LISTENING状态，开始接收音频

### 3. 硬打断机制

**触发条件**:
- 当前状态: is_playing_tts=True（数字人正在说话）
- 检测到唤醒词，置信度达到阈值：
  - 视觉唤醒路径: ≥ 0.6
  - 纯语音唤醒路径: ≥ 0.9

**处理流程**:
1. 清空TTS播放队列和预缓冲区
2. 生成新的session_id，屏蔽旧session的音频帧
3. 发送client_interrupt消息给LLM Agent
4. 启动冷却期（忽略VAD和KWS事件）
5. 设置is_playing_tts=False，保持is_interactive_mode=True（继续交互）

**实例**:
- 数字人正在播放"今天天气很好..."
- 用户说"你好小康"（置信度65%，视觉唤醒路径）
- 预期: TTS立即停止，播放队列清空，系统进入LISTENING状态

### 4. 视觉斩断机制

视觉斩断是最高优先级的打断机制，当检测到用户离开摄像头时立即切断交互。

**触发条件**:
- vision_wake == False（视觉唤醒状态机判定用户已离开）
- visual_cutoff_enabled == True

**场景A: 交互模式下的视觉斩断（关麦）**

触发条件:
- 当前状态: is_interactive_mode=True, is_playing_tts=False
- is_vision_target_present == False（所有人脸都不在[0.4m, 4.5m]区间内，经过防抖窗口）

处理:
- 如果存在未完成的对话（_current_trace_id不为None），先发送audio_end消息闭合ASR管道
- 发送timeout_exit消息给LLM Agent
- 停止音频推流
- 退出交互模式（is_interactive_mode=False）

**场景B: 播放状态下的视觉斩断（停止TTS）**

触发条件:
- 当前状态: is_interactive_mode=True, is_playing_tts=True
- is_vision_target_present == False（经过防抖窗口）

处理:
- 发送client_interrupt消息给LLM Agent（reason: visual-cutoff）
- 清空播放队列和预缓冲区
- 生成新的session_id
- 退出交互模式（is_interactive_mode=False, is_playing_tts=False）

**实例**:
- 用户在LISTENING状态下说话，然后离开摄像头
- 预期: 系统立即停止音频推流，发送ABORT_SPEECH，返回IDLE状态

### 5. VAD超时机制（纯事件驱动）

VAD超时机制采用纯事件驱动架构，在每个音频帧到达时进行检查，避免多线程锁竞争。

#### 宏观超时（发呆检测）

**触发条件**:
- 用户未开口（_user_has_spoken == False）
- 距离上次VAD检测时间 > 5.0秒（vad_silence_timeout_default_sec）
  - 纯语音唤醒模式: 5秒无语音
  - 视觉降维打击模式: 5秒无语音且无唇动（经过防抖）

**处理**:
- 发送audio_end消息给LLM Agent（如果存在trace_id）
- 停止音频推流
- 退出交互模式（is_interactive_mode=False）

**实例**:
- 用户唤醒后5秒内没有说话
- 预期: 系统退出交互模式，返回空闲状态

#### 微观超时（说话结束检测 - 双模态快刀）

**触发条件**:
- 用户已开口（_user_has_spoken == True）
- time_since_last_vad >= 1.5秒 且 time_since_lip_closed >= 1.5秒（连续无声音且无唇动）

**处理**:
- 发送audio_end消息给LLM Agent（trace_id）
- 清空trace_id
- 停止音频推流
- 保持交互模式（is_interactive_mode=True），等待LLM回复

**实例**:
- 用户说完话，音频VAD静音1.5秒，视觉检测嘴巴闭合1.5秒
- 预期: 系统触发双模态快刀，发送audio_end，等待LLM回复

**架构优势**:
- 纯事件驱动：在每个音频帧到达时检查，无需独立线程
- 单线程安全：避免多线程锁竞争，特别是对_current_trace_id的读写
- 实时性更好：每个音频帧（20ms或100ms）都会检查，响应更快

### 6. 30秒保底机制

**触发条件**:
- 当前状态: is_interactive_mode=True
- 进入交互模式后已持续30秒

**处理**:
- 强制发送audio_end消息（如果存在trace_id）
- 停止音频推流
- 退出交互模式（is_interactive_mode=False）

**目的**: 防止在视觉受损或极度噪音环境下，其他截断机制失效导致数据无限制传输

**实例**:
- 用户在极度噪音环境下说话，VAD和视觉检测都失效
- 预期: 30秒后系统强制截断，退出交互模式

### 7. 90秒交互超时机制

**触发条件**:
- 当前状态: is_interactive_mode=True
- 自上次VAD从True变为False后，已持续90秒无交互

**处理**:
- 退出交互模式（is_interactive_mode=False）
- 恢复默认阈值（0.9）
- 允许立即再次唤醒（无需冷却期）

**目的**: 防止系统长时间处于交互模式，消耗资源

**实例**:
- 用户唤醒后，长时间没有说话（VAD一直为False）
- 预期: 90秒后系统自动退出交互模式，返回空闲状态

### 8. 持续对话机制

**触发条件**:
- TTS播放完成，自动从is_playing_tts=True转换到is_playing_tts=False
- is_interactive_mode保持True

**流程**:
1. 设置is_playing_tts=False
2. 激活0.8秒物理防抖护盾（屏蔽麦克风瞬间开启时的物理电流声）
3. 发送start_streaming命令给音频模块（开启持续拾音）
4. 保持is_interactive_mode=True，等待用户继续说话

**实例**:
- 数字人回答完成，系统自动继续交互模式
- 用户在免唤醒窗口内继续说话
- 预期: 系统继续接收音频，无需再次唤醒

## 数据流链路

### 完整对话链路

```
用户说话
  ↓
音频模块 (KWS + VAD)
  ↓ ZMQ PUB (Multipart: JSON + PCM)
Core Server
  ↓ WebSocket (文本帧: audio_start + trace_id)
  ↓ WebSocket (二进制帧: PCM音频流，带trace_id)
LLM Agent服务器 (含ASR和TTS)
  ↓ WebSocket (文本帧: route/stop_tts/TTS_DONE/interrupt)
  ↓ WebSocket (二进制帧: PCM音频流，带session_id)
Core Server
  ↓ 音频播放队列（预缓冲机制，100ms）
sounddevice → XVF3800扬声器
```

### 视觉数据链路

```
摄像头
  ↓
视觉模块 (MediaPipe)
  ↓ ZMQ PUB (JSON)
Core Server
  ↓
视觉唤醒状态机
  ↓
视觉斩断判断
```

### 控制指令链路

```
Core Server
  ↓ ZMQ REQ-REP
音频模块
  - set_threshold: 动态调整唤醒词阈值（0.9 ↔ 0.6）
  - start_streaming: 开启音频推流（自动回捞800ms历史音频）
  - stop_streaming: 停止音频推流
  - reset_cooldown: 重置冷却期

Core Server
  ↓ ZMQ PUB
视觉模块
  - START_LIP_SYNC: 启动口型同步
  - STOP_LIP_SYNC: 停止口型同步

Core Server
  ↓ WebSocket (文本帧)
LLM Agent服务器
  - audio_start: 开始音频流（带trace_id）
  - audio_end: 结束音频流（带trace_id和reason）
  - client_interrupt: 客户端打断信号（硬打断或视觉斩断）
  - device_state: 设备状态上报（idle/listening/thinking/speaking）
  - timeout_exit: 超时退出（视觉斩断时）

LLM Agent服务器
  ↓ WebSocket (文本帧)
Core Server
  - route: TTS文本（仅用于状态上报）
  - stop_tts: 停止TTS合成
  - TTS_DONE: TTS合成完成
  - interrupt: 服务端打断指令（带session_id）

## 典型场景实例

### 场景1: 完整对话流程（视觉唤醒路径）

**步骤**:
1. 用户走到摄像头前（距离2.5m，正面率80%）
2. 系统进入VISUAL_WAKE状态，阈值降至0.4
3. 用户说"你好小康"（置信度60%）
4. 系统进入LISTENING状态，开始接收音频
5. 用户说"今天天气怎么样"
6. 用户说完，音频VAD静音1.0秒，视觉检测嘴巴闭合1.0秒
7. 系统进入PROCESSING状态，发送END_OF_SPEECH
8. ASR识别完成，发送文本给LLM
9. LLM生成回复，发送给TTS
10. TTS合成音频，发送给Core Server
11. Core Server进入SPEAKING状态，播放音频
12. 播放完成，系统进入LISTENING状态（持续对话）

**预期结果**:
- 状态转换: IDLE → VISUAL_WAKE → LISTENING → PROCESSING → SPEAKING → LISTENING
- 唤醒路径: visual
- 对话轮次: 0 → 1

### 场景2: 硬打断测试

**步骤**:
1. 系统处于SPEAKING状态，正在播放"今天天气很好，适合出门散步..."
2. 用户在播放过程中说"你好小康"（置信度65%，视觉唤醒路径）
3. 系统检测到唤醒词，触发硬打断

**预期结果**:
- TTS立即停止合成
- 播放队列被清空
- 系统进入LISTENING状态
- 启动1.5秒冷却期
- 日志显示: "硬打断：停止播报，清空队列，发送停止信号"

### 场景3: 视觉斩断测试

**步骤**:
1. 系统处于LISTENING状态（视觉唤醒路径，conversation_round=1）
2. 用户正在说话
3. 用户离开摄像头（距离>3.5m，持续1.5秒）
4. 视觉唤醒状态变为False

**预期结果**:
- 系统立即停止音频推流
- 发送ABORT_SPEECH标记给ASR
- 状态回落到IDLE
- 音频阈值恢复为0.95
- 日志显示: "视觉斩断：检测到用户离开，立即切断监听"

### 场景4: 纯语音唤醒首次对话

**步骤**:
1. 用户不在摄像头前（或距离>3m）
2. 用户说"你好小康"（置信度96%）
3. 系统进入LISTENING状态
4. 用户说话，然后离开摄像头

**预期结果**:
- 唤醒路径: audio
- 首次对话（conversation_round=0）不触发视觉斩断
- 允许完成首次对话，即使用户离开摄像头

### 场景5: 宏观发呆超时

**步骤**:
1. 系统进入LISTENING状态
2. 用户唤醒后2秒内没有说话（_user_has_spoken == False）
3. 系统检测到宏观超时

**预期结果**:
- 如果vision_wake == True: 进入VISUAL_WAKE状态
- 如果vision_wake == False: 进入IDLE状态
- 发送ABORT_SPEECH标记（用户未开口）
- 日志显示: "检测到幽灵唤醒（用户未开口），强制丢弃音频"

### 场景6: 30秒保底机制

**步骤**:
1. 系统进入LISTENING状态
2. 在极度噪音环境下，VAD和视觉检测都失效
3. 系统持续接收音频30秒

**预期结果**:
- 30秒后系统强制截断
- 发送END_OF_SPEECH标记
- 进入PROCESSING状态
- 日志显示: "30秒交流上限时间已到，强制截断"

## 配置参数说明

### 视觉唤醒配置

```yaml
vision_wake:
  detection_distance_m: 4.0        # 唤醒检测距离（米）
  frontal_percent_threshold: 70    # 正面率阈值（%）
  distance_range: [0.4, 4.5]       # 保持唤醒的有效距离区间（米），用于视觉斩断判断
  enter_debounce_ms: 200           # 进入防抖窗口（毫秒），要求极速响应
  leave_debounce_ms: 1000          # 离开防抖窗口（毫秒），防止误判
  visual_cutoff_enabled: true     # 启用视觉斩断机制
```

### 音频阈值配置

```yaml
audio_threshold:
  default: 0.9      # 默认高阈值（无人时，纯语音唤醒）
  visual_wake: 0.6  # 视觉唤醒后的低阈值（视觉降维打击）
```

### 持续对话配置

```yaml
conversation:
  vad_silence_timeout_default_sec: 5.0  # 宏观超时（秒）：纯语音唤醒模式下，5秒无语音则结束对话
  vad_silence_timeout_visual_sec: 5.0   # 宏观超时（秒）：视觉降维打击模式下，5秒无语音且无唇动则结束对话
  vad_fast_cutoff_sec: 1.5              # 双模态快刀超时（秒）：连续1.5秒无声音且无唇动则结束对话
  interactive_timeout_sec: 90.0         # 交互超时（秒）：90秒无交互则退出交互模式
```

### VAD配置

```yaml
vad:
  engine: "silero"           # VAD引擎
  model_version: "v4"        # Silero VAD版本
  threshold: 0.5             # 语音概率阈值
  sample_rate: 16000        # 采样率
```

### 音频播放配置

```yaml
audio_playback:

  - 如果config.yaml中设置为16000，脚本中保持 `SAMPLE_RATE = 16000`

### 截断策略阈值

- **双模态快刀**: 音频静音1.5秒 且 嘴巴闭合1.5秒（连续无声音且无唇动）
- **宏观超时（纯语音）**: 无语音5.0秒（用户未开口或已开口后）
- **宏观超时（视觉降维）**: 无语音且无唇动5.0秒（经过防抖）
- **30秒保底**: 持续监听30秒（最高优先级，防止数据无限制传输）
- **90秒交互超时**: 自上次VAD从True变为False后，90秒无交互则退出交互模式

## 关键设计要点

### 1. TraceID管理

- **生成时机**: 只有在交互模式下，trace_id==None，且VAD从False变True时生成
- **用途**: 标记每轮对话，用于LLM Agent的精确对话定位
- **绑定**: traceID与数据在传输时绑定（WebSocket消息中携带）
- **清空**: 发送audio_end后清空，或视觉斩断时先发送audio_end再清空

### 2. SessionID管理

- **生成时机**: 收到第一个TTS音频帧时生成，或硬打断/视觉斩断时重新生成
- **用途**: 在TTS音频播放时过滤旧音频帧，防止残留播放
- **机制**: 如果收到旧session_id的二进制帧，直接在接收端丢弃

### 3. 音频预回滚机制

- **触发**: 生成新traceID时，发送start_streaming命令
- **实现**: audio_service自动从Ring Buffer中回捞800ms历史音频（pre_roll_ms配置）
- **方式**: 异步管道（audio_service自己推入PUB通道），不阻塞core_server

### 4. 纯事件驱动架构

- **VAD超时检查**: 在每个音频帧到达时检查，无需独立线程
- **优势**: 避免多线程锁竞争，特别是对_current_trace_id的读写
- **实时性**: 每个音频帧（20ms或100ms）都会检查，响应更快

### 5. 90秒交互超时

- **触发**: 每次VAD由True变False时，重置并重新开始90秒倒计时
- **超时**: 90秒后退出交互模式（is_interactive_mode=False）
- **恢复**: 恢复默认阈值（0.9），允许立即再次唤醒

### 6. 冷却期机制

硬打断后启动冷却期，期间忽略VAD和KWS事件，防止误触发。

## 故障排查要点

### 问题1: 唤醒词检测不到

- 检查音频模块是否正常运行
- 检查阈值配置（default: 0.95, visual_wake: 0.4）
- 检查唤醒路径是否正确（visual路径使用0.4阈值）

### 问题2: 视觉斩断不生效

- 检查visual_cutoff_enabled是否为true
- 检查视觉唤醒状态机是否正确维护vision_wake
- 检查是否在保护场景（纯语音唤醒首次对话、首次回答）

### 问题3: VAD超时不触发

- 检查_user_has_spoken是否正确设置
- 检查VAD检查线程是否正常运行
- 检查时间戳是否正确更新

### 问题4: 硬打断不生效

- 检查唤醒词置信度是否达到阈值
- 检查唤醒路径（visual: 0.4, audio: 0.95）
- 检查TTS模块是否收到STOP_SYNTHESIS信号

### 问题5: ASR识别出"嗯"

- 检查是否在非LISTENING状态下发送了END_OF_SPEECH
- 检查是否在用户未开口时发送了END_OF_SPEECH（应发送ABORT_SPEECH）

## 总结

WakeFusion系统通过多模态融合、状态机管理、智能截断等机制，实现了稳定可靠的数字人交互系统。核心设计包括：

1. **双重唤醒路径**: 视觉降维打击（低阈值）和纯语音唤醒（高阈值）
2. **智能截断机制**: 宏微观双重监控、视觉强杀、30秒保底
3. **硬打断机制**: 支持在数字人说话时立即打断
4. **视觉斩断机制**: 最高优先级打断，用户离开立即切断
5. **持续对话机制**: 免唤醒窗口，提升交互流畅度

系统通过ZMQ和WebSocket实现模块间通信，确保低延迟和高可靠性。
