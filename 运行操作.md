建议你按照**“从局部到整体”**的顺序进行测试：

1. 硬件探测（找设备）
首先确认软件能不能看到你的麦克风。

Bash
python -m tests.list_audio_devices
操作：运行后观察控制台打印的列表。

结果：你应该能看到带有 "XMOS" 或 "XVF3800" 字样的设备索引号。

2. 音频驱动测试（听声音）
验证驱动层是否能持续获取 16kHz 的音频数据流 。

Bash
python -m tests.test_audio_driver
操作：对着麦克风说话，或播放之前提到的 MP3 模拟素材。
脚本通过四个独立的测试函数，自底向上地验证了整个音频链路：

（1） 音频设备枚举测试 (test_audio_devices)
测试目的：验证系统是否能“看见”麦克风硬件。
工作原理：它会扫描电脑上所有的录音设备，寻找名称匹配 "XVF3800" 的设备索引。
关键指标：如果显示“找到设备”，说明 USB 连接和 Windows 驱动正常；如果显示“未找到”，说明硬件未识别。

（2） 实时音频采集测试 (test_audio_capture)
测试目的：验证数据传输的稳定性和质量。
工作原理：开启麦克风录音 5 秒，每秒应接收约 50 帧音频。
关键指标：
RMS能量：反映音量大小，数值波动说明采集到了真实声音。
实际FPS：如果远低于 50.0，说明系统存在丢帧或卡顿。

（3） 唤醒词识别测试 (test_kws_model)
测试目的：验证 openWakeWord 模型是否能正常加载并进行 AI 推理。
工作原理：加载训练好的唤醒模型，并向其输入一段测试音频。
关键点：这正是你之前报错的地方。它不仅测试模型文件是否存在，还测试环境中的推理引擎（如 ONNX 或 TFLite）是否已装好。

（4） 语音活动检测测试 (test_vad_model)
测试目的：验证系统能否区分“环境噪音”和“人说话声”。
工作原理：使用 webrtcvad 库对音频片段进行分析。
作用：这是数字人节省算力的关键——只有当 VAD 判断有人在说话时，系统才会去运行更复杂的语音识别（STT）。

3. 唤醒词模型测试（测灵敏度）
验证 MatchboxNet 模型在不联网的情况下是否工作 。

Bash
python tests/test_matchboxnet_kws.py --input test_audio.wav
操作：使用你准备好的“带噪声的唤醒词”音频文件作为输入。


结果：程序会输出 Confidence Score（置信度）。如果分数接近 1.0，说明识别极准 。

4. 多模态集成测试（测逻辑）
这是最关键的一步，测试“音频触发 + 视觉验证”的完整逻辑。

Bash
python tests/test_matchboxnet_microphone.py

操作：开启程序后，你先站在相机外说唤醒词，再站在相机前（1-5米）说唤醒词 。

结果：观察控制台是否输出了 [WAKE_CONFIRMED] 事件。如果只有音频触发但没有视觉确认（人脸或距离不达标），系统不应产生唤醒确认。

1. 退出当前环境（极其重要）
你不能拆掉你正坐在里面的房子。所以必须先退回到 base 环境：

Bash
conda deactivate
2. 彻底删除被“玩坏”的环境
毫不留情地把原来那个被污染的 wakefusion 删掉：

Bash
conda env remove -n wakefusion
3. 将备份环境“扶正”（重命名）
如果你使用的是较新版本的 Conda（大部分人都是），直接使用重命名命令：

Bash
conda rename -n wakefusion_bak wakefusion
(💡 备用方案：如果你的 Conda 版本太老，报 rename 命令不存在，那就再克隆回来：conda create -n wakefusion --clone wakefusion_bak，然后再把 wakefusion_bak 删掉即可。)

4. 满血复活，重新激活！
